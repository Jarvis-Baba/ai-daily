## 1. 设计自审

以首席产品负责人 + 首席研究员的身份，对 v2.0 做无保留审查。

### 1.1 可能错误的设计假设

| 假设 | 为什么可能错 | 严重程度 |
|------|-------------|----------|
| "55 篇文章足以覆盖当日重要事件" | RSS 源存在系统性偏见：英语为主、科技媒体为主、美国中心。中文 AI 生态（微信、知乎、机器之心）完全盲区 | 高 |
| "LLM 能可靠地打出 1-10 分数" | LLM 数值评分存在：中位数偏好（挤在 5-7 分）、时间不一致性（同一文章两次评分可差 2-3 分）、锚定效应（前序文章影响后续评分） | 高 |
| "影响范围可以在生成时判断" | 部分事件的影响需要数月才能显现。生成时的 Impact 评分本质上是"感知影响"而非"真实影响" | 中 |
| "Gamma = 丢弃，不会后悔" | 存在"睡美人"问题——今天看起来是噪音的事件，明天可能因为关联事件而变成信号。没有恢复机制 | 中 |
| "读者每天只有 3-5 分钟" | 如果日报确实提供了决策价值，读者会花更多时间。不应为了缩短而牺牲深度。约束应该是"信息密度"而非"总字数" | 低 |

### 1.2 无法在生成时真实判断的指标

- **不可逆性**：需要事后验证。删除此维度。
- **真实新颖度**：受限于我们的信息源覆盖范围。对我们是新的，对行业可能是旧闻。
- **长期可行动性**：基础设施/监管类事件需要数月才产生行动含义，生成时容易被低估。

### 1.3 看似合理但不可执行的规则

| 规则 | 问题 |
|------|------|
| "Alpha 必须三条全中" | "不可逆"在生成时无法判断，导致 Alpha 判定永远缺一条腿 |
| "日报总计 ≤800 字" | LLM 输出长度不可精确控制。应改为结构约束（Alpha ≤120 字、Beta ≤50 字）加后处理裁剪 |
| "每条行动必须可独立执行" | 有些真正重要的行动是"观察并等待信号"。不应把所有非即时行动判为不合格 |
| "所有 Gamma 直接丢弃" | 没有定义"接近 Beta 的 Gamma"如何处理。需要缓冲区 |

### 1.4 高置信度错误判断的风险点

1. **今日判断**：日报开篇的一句话判断代表全天判断。如果错，整份日报的可信度归零。需要约束 LLM 只下"有证据支持"的判断。
2. **行动建议**：从融资新闻推导"你应该创业"、从模型升级推导"你应该重构系统"——这是最危险的模式。缺乏证据链条的推导必须禁止。
3. **评分数字的权威假象**：LLM 输出的分数看起来精确、客观，但实际上不稳定。需要在系统层面标注"评分由 AI 生成，仅供参考"。
4. **单向叙事**：如果当天所有 Alpha/Beta 都在说"增长""突破""加速"，读者会形成系统性乐观偏见。

## 2. Alpha 重定义

### 2.1 旧版定义的问题

```
旧版 Alpha = 影响范围广 + 不可逆 + 可行动（三条全中）
```

核心漏洞：**"不可逆"需要预知未来**。

2023 年的 AutoGPT、Humane AI Pin 在当时都会被判定为"不可逆的结构性变化"。事后证明不是。日报系统不应该假装自己能预测不可逆性。

### 2.2 新版 Alpha 定义

```
Alpha = 改变至少一项行业默认假设的事件

判据（三条全中）：
1. 影响广度 — 影响 ≥30% 的目标受众决策域
2. 新颖密度 — 不能从上周的趋势线外推得出
3. 决策相关性 — 今天就可以据此改变至少一个具体决策
```

### 2.3 关键变化

| 旧版 | 新版 | 原因 |
|------|------|------|
| 不可逆 | 新颖密度 | 不可逆需要未来知识。新颖密度可在生成时判断：和 Theme Memory 比对，这件事是否偏离了已有趋势线 |
| 可行动 | 决策相关性 | 原版暗示"明天就能动手做"。新版承认"改变决策"本身也是行动——包括决定观察、决定暂停、决定加仓 |

### 2.4 Alpha 判据的可操作化

**影响广度怎么判断（生成时）**：
- 该事件是否涉及 Top5 AI 公司（OpenAI、Anthropic、Google、Meta、Microsoft）？
- 是否涉及跨赛道影响（不止影响模型层，还影响应用层/工具层/芯片层）？
- 是否涉及监管或政策变化？
- 任意两项命中 → Impact ≥ 8 的可能性高

**新颖密度怎么判断（生成时）**：
- 与 Theme Memory 比对：该主题过去 7 天是否出现过？
- 如果出现过，今天的增量信息是否显著改变了该主题的方向或强度？
- 如果未出现过，是否为领域首次出现的事件类型？
- 与已有主题显著不同 + 增量信息改变方向/强度 → Novelty ≥ 7

**决策相关性怎么判断（生成时）**：
- 如果这条新闻为真，一个 AI 独立开发者今天会不会做出与昨天不同的选择？
- "不同选择"包括：开始学某个技术、停止用某个工具、加速推进某个方向、重新评估某个假设
- 如果能写出至少一个具体决策变化 → 决策相关性成立

### 2.5 Alpha 数量规则

- 硬上限：2 条
- 没有符合判据的：**空着**。不凑数
- 如果只有 1 条：不强制填满 2 条
- 如果 0 条：今日判断写"今天没有改变基础假设的变量"，这本身就是有价值的信号

## 3. 评分体系重构

### 3.1 乘法模型的问题

旧版：`Alpha 分 = Impact × Novelty × Actionability`

**问题一：过度惩罚高 Impact 低 Actionability 的事件**

美国 AI 监管法案：Impact=10, Novelty=9, Actionability=3 → 270 分
某 API 开放：Impact=7, Novelty=6, Actionability=9 → 378 分

API 开放得分高于监管法案。这不合理。

**问题二：Actionability 隐性权重过高**

三项相乘，Actionability 低可以直接把 Impact 和 Novelty 拉下水。对于基础设施、基础模型、监管类事件——它们天然 Actionability 低但长期决定行业格局——乘法模型系统性低估。

**问题三：分数分布不可控**

乘积（1-1000）的分布极不均匀。中间分数段（100-500）挤了大量内容，难以设定清晰的桶边界。

### 3.2 新版：加权利差模型

```
总分 = 0.5 × Impact + 0.3 × Novelty + 0.2 × Actionability

分数范围：1.0 – 10.0
```

**权重设计原理**：
- **Impact 占 50%**：日报的核心价值是"什么重要"。影响力是第一排序维度
- **Novelty 占 30%**：新颖度决定"是否需要改变认知"。重复信息即使 Impact 高也不值得在日报中占据 Alpha 位置
- **Actionability 占 20%**：行动价值是加分项，不是硬门槛。高 Impact 事件即使不能立即行动也必须出现在日报中

### 3.3 分桶规则

```
Alpha 候选：Impact ≥ 8 且 Novelty ≥ 7
           （Actionability 不作为 Alpha 门槛，只影响总分）

Beta 候选：总分 ≥ 5.0 或 Impact ≥ 6

Gamma：     总分 < 5.0 且 Impact < 6
```

**Alpha 不计 Actionability 门槛的原因**：
如果美国 AI 监管法案出台（Impact=10, Novelty=9, Actionability=3），乘法模型给它 270 分，可能掉出 Alpha。加法模型给它 5.0 + 2.7 + 0.6 = 8.3 分，且满足 Impact≥8 + Novelty≥7，稳稳进入 Alpha。这才是对的。

### 3.4 排序规则

日报内排序原则：

1. Alpha 永远在 Beta 之前
2. Alpha 之间按总分降序
3. Beta 之间按总分降序
4. 如果某 Beta 的 Actionability=9 或 10（即"明天就能行动"），允许它在 Beta 列表中置顶标注

### 3.5 评分流程的两轮设计

```
第一轮（FilterStage）：
  输入：55 篇标题+摘要
  评分：Impact(1-10) + Novelty(1-10)
  筛选：Impact+Novelty ≥ 12 → 15-20 篇进入下一轮
  目的：不浪费全文抓取在明显 Gamma 上

第二轮（ScoringStage）：
  输入：15-20 篇全文
  评分：Impact(1-10) + Novelty(1-10) + Actionability(1-10)
  计算：加权总分
  分桶：Alpha / Beta / Gamma
  目的：有充分信息做精确判断
```

## 4. 行动建议风险控制

### 4.1 问题陈述

LLM 存在 "建议过拟合" 倾向：从有限证据推导出过度激进的结论。

典型危险模式：
- 融资新闻 → "你应该去这个赛道创业"
- 模型升级 → "你应该重构系统架构"
- 某公司裁员 → "你应该学 X 技能转行"
- 某技术流行 → "你应该立即采用 X"

这些建议的共同特征：**证据链条断裂**。从单点新闻到具体行动之间隔着多层假设，LLM 直接跳过了。

### 4.2 Action Framework：三级行动体系

#### Level 1 — 直接行动（可输出）

**定义**：新闻中含明确、可验证的使能信息，读者今天就可以采取行动。

**判据（四条全中）**：
1. 行动直接来自新闻中陈述的事实（不是推论）
2. 行动有明确的产出或可验证的结果
3. 行动不需要读者改变职业方向、技术栈根基或资金结构
4. 存在至少一条源文章直接支撑该行动

**格式**：`{动词} {对象} {产出/目的}（支撑：{源文章标题}）`

**示例**：
- "注册 Groq Cloud 免费额度，测试 Llama-4 在推理场景下的延迟（支撑：Groq 融资 6.5 亿扩大推理云）"
- "将 Claude Opus 4.8 加入模型评估 pipeline，对比上一版在 honesty 维度的差异（支撑：Claude Opus 4.8 发布）"

#### Level 2 — 观察行动（可输出，但标注为观察）

**定义**：新闻改变了某件事的概率分布，但尚未产生可立即执行的行动。读者应建立观察机制。

**判据**：
1. 行动是"建立监测"而非"采取执行动作"
2. 有明确的观察窗口和触发条件

**格式**：`观察 {对象}，时间窗口 {X周/月}。触发条件：{具体信号}。如果触发，则 {行动}（支撑：{源文章标题}）`

**示例**：
- "观察 Anthropic IPO 进展，时间窗口 3 个月。触发条件：S-1 文件公开。如果触发，评估 AI 基础设施板块持仓（支撑：Anthropic $65B 融资估值近万亿）"
- "观察美国 OMB 科研资助新规进展，时间窗口 2 个月。触发条件：正式生效或联邦法院介入。如果触发，排查涉及美国科研合作的项目合规风险（支撑：OMB 新规提案）"

#### Level 3 — 禁止输出

**定义**：以下类型的"建议"在任何情况下都不输出。

**禁止清单**：
- 建议读者创业、转型、换工作、重构系统（涉及职业/资金/架构的根本改变，单条新闻不足以支撑）
- 没有源文章直接支撑的推论性建议
- 使用"应该""必须""最好"等确定性表述但没有相应确定性证据的建议
- "如果是我，我会……"类型的个人观点伪装成建议
- 涉及具体投资标的、仓位、时机的建议（日报不是投资顾问）

### 4.3 行动输出的质量控制

每条行动在输出前必须通过以下检查：

```
□ 这条行动有至少一条源文章直接支撑吗？
  （不是推论，而是文章中明确提到的事实）
□ 读者读完行动和源文章后，能独立判断该行动是否适用于自己吗？
□ 行动是 Level 1（直接）还是 Level 2（观察）？
  （如果是 Level 3 范畴，直接删除）
□ 是否使用了"应该""必须""最好"等表述？
  （如果是 Level 2，改为"观察""监测""评估"）
□ 如果行动错了，读者会遭受不可逆的损失吗？
  （如果是，降级或删除）
```

## 5. Counter Signal：反向信号

### 5.1 定义

**Counter Signal**：当日新闻中，与主流叙事方向相反或构成摩擦的事件/观点。

### 5.2 为什么重要

AI 媒体生态存在结构性乐观偏见：
- 融资、产品发布、模型升级 → 天然被放大
- 失败、质疑、阻力 → 天然被淡化
- 日报如果只呈现"什么在增长"，会系统性高估行业乐观度

Counter Signal 的价值不是"唱反调"，而是**帮读者看到被主流叙事掩盖的摩擦面**。

### 5.3 识别方法

在 SynthesizeStage 中执行：

1. **提取当日主流叙事**：从 Alpha + Beta 中归纳出 1-2 个主导方向（例如"Agent 生态扩张""资本加速流入 AI""模型可靠性提升"）
2. **扫描反向证据**：在 Gamma 和未入选文章中搜索与主流叙事方向相反或增加重要限制条件的信号
3. **可信度过滤**：反向信号必须有可验证的事实依据（不是观点文章，不是个人博客猜测）
4. **只输出有张力的信号**：如果反向证据太弱，不强制输出

### 5.4 展示格式

```
## ⚡ 反向信号

**主流叙事**：{当日 Alpha + Beta 的归纳方向}

**反向信号**：{标题} — {如何与主流叙事形成张力}

**为什么值得注意**：{一句话——这个摩擦面为什么可能重要}

*{来源}*
```

### 5.5 约束

- 每天最多 1 条 Counter Signal
- 如果没有可信的反向证据，**不输出此板块**
- Counter Signal 不是"负面新闻汇总"，而是"与主流叙事的摩擦点"
- 禁止为了"看起来客观"而强行制造对立

## 6. Theme Memory：主题记忆

### 6.1 问题

当前日报每天独立生成，没有记忆。Anthropic 融资和 Claude 更新被视为两个孤立事件，而不是同一主题的两次出现。读者无法从日报中感知趋势的持续性和强度变化。

### 6.2 设计

Theme Memory 是一个持久化的 JSON 文件，追踪跨日期的主题演变。

**存储位置**：`output/.theme-memory.json`

**数据结构**：

```json
{
  "version": "1.0",
  "updated": "2026-05-31",
  "themes": {
    "anthropic_ecosystem": {
      "name": "Anthropic 生态系统扩张",
      "first_seen": "2026-05-12",
      "last_seen": "2026-05-31",
      "mention_count": 14,
      "trend": "accelerating",
      "intensity": 8,
      "key_events": [
        {"date": "2026-05-12", "event": "Claude Opus 4.7 发布"},
        {"date": "2026-05-28", "event": "$65B 融资，估值近万亿"}
      ],
      "related_themes": ["ai_capital_flow", "agent_tooling"]
    },
    "ai_infra_shift_to_inference": {
      "name": "AI 基础设施从训练转向推理",
      "first_seen": "2026-04-20",
      "last_seen": "2026-05-31",
      "mention_count": 8,
      "trend": "sustained",
      "intensity": 7,
      "key_events": [
        {"date": "2026-04-20", "event": "AWS 推出推理优化实例"},
        {"date": "2026-05-29", "event": "Groq 融资 6.5 亿拓展推理云"}
      ],
      "related_themes": ["ai_chip_competition"]
    }
  }
}
```

**字段说明**：

| 字段 | 含义 | 取值 |
|------|------|------|
| `trend` | 主题强度方向 | `accelerating` / `sustained` / `decelerating` / `dormant` |
| `intensity` | 主题当前热度（1-10） | 基于最近 7 天提及频率 + Alpha/Beta 占比 |
| `key_events` | 里程碑事件 | 只保留该主题下曾入选 Alpha 或高 Impact Beta 的事件 |

### 6.3 更新规则

每次日报生成后，ThemeStage 执行：

1. **匹配**：将今日 Alpha + Beta 与已有主题做语义匹配（可通过 LLM 判断是否属于同一主题，也可以通过关键词/源/公司名做简单匹配）
2. **更新**：匹配成功 → 更新 `mention_count`、`last_seen`、`intensity`、`key_events`（如有新里程碑）
3. **新建**：不匹配任何已有主题且 Impact ≥ 7 → 创建新主题
4. **衰减**：超过 30 天未出现的主题 → `trend` 标记为 `dormant`，intensity 减半
5. **合并**：两个主题 `key_events` 重叠 ≥50% → LLM 判断是否合并为一个主题

### 6.4 如何提升日报价值

在 SynthesizeStage 中，Theme Memory 提供三个能力：

1. **连续性标注**：Beta 条目可附带主题轨迹。例如：
   > Anthropic 估值近万亿（该主题第 14 次出现，趋势持续增强）
2. **趋势判断**：如果在 Theme Memory 中某主题从 `sustained` 变为 `accelerating`，可在"结构趋势"板块提及加速信号
3. **Counter Signal 辅助**：如果某主题一直 `accelerating`，但今天出现相反证据 → 自动成为 Counter Signal 候选

### 6.5 限制

- Theme Memory 提升的是"叙事连续性"，不能替代真实因果分析
- 语义匹配可能出错（两个不相关事件被归为同一主题）
- 冷启动：前 7 天主题检测不稳定，需要积累
- 主题分裂：一个主题可能分化为两个子主题。当前设计只能手动或靠 LLM 判断，自动化程度有限

## 7. 最终架构：v2.1 完整管线

### 7.1 管线总览

```
RSS Feed (7 sources, ~55 articles)
  │
  ▼
┌──────────────────────────────────────────────┐
│ FilterStage                                   │
│ 作用：快速初筛，节省全文抓取成本              │
│ 输入：55 篇（标题 + 摘要 + 来源 + 链接）      │
│ 评分：Impact(1-10) + Novelty(1-10)           │
│ 输出：15-20 篇（Impact+Novelty ≥ 12）        │
│ 丢弃：~35 篇明显 Gamma                        │
└──────────────────────────────────────────────┘
  │
  ▼
┌──────────────────────────────────────────────┐

│   2. 为每条 Alpha 生成判断+变量+行动          │
│   3. 为每条 Beta 生成一句话要点               │
```

## 8. 验收自评

### 8.1 v2.1 解决了什么

| v2.0 问题 | v2.1 解决方案 | 效果 |
|-----------|--------------|------|
| "不可逆"无法判断 | 替换为"新颖密度"，与 Theme Memory 比对 | Alpha 判定可以在生成时完成 |
| Actionability 权重过高 | 改为加法模型，Actionability 占 20% | 高 Impact 低 Actionability 事件不会被漏掉 |
| 乘法模型分布失控 | 加法模型 1.0-10.0 线性分布 | 桶边界清晰 |
| 行动建议可能激进 | Action Framework 三级体系 + 输出前检查清单 | Level 3（创业/转型/重构）被硬阻断 |
| 单向叙事 | Counter Signal 板块 | 读者能看到与主流叙事相反的信号 |
| 每天独立，没有连续性 | Theme Memory 持久化 + ThemeStage | 日报可引用主题轨迹和历史强度 |
| Gamma 丢弃无后悔药 | 近 Beta 的 Gamma 保留 1 天缓冲 | 降低"睡美人"信息丢失风险 |
| LLM 评分不可靠 | 标注"AI 生成，仅供参考" + 两轮评分降低噪声 | 降低权威假象风险 |

### 8.2 v2.1 仍然无法解决的问题

| 局限 | 为什么无法解决 | 可能的缓解措施 |
|------|---------------|---------------|
| **RSS 源系统性偏见** | 受限于可用的英文 RSS 源。中文 AI 生态（微信公众号、知乎、机器之心）因反爬和内容格式问题无法自动抓取 | 手动添加高价值中文源链接；长期考虑 WeChat RSS 桥接 |
| **LLM 评分不一致** | LLM 对数值评分的固有噪声无法通过 prompt 工程消除。同一文章两次评分差 2-3 分是常态 | 接受噪声。不把分数当精确值，只用于排序和分桶。不向用户展示具体分数 |
| **新颖度受限于知识边界** | 系统只知道 Theme Memory 中记录的历史。一个在行业里已知但在 Theme Memory 中未出现的事件会被高估新颖度 | Theme Memory 积累时间越长越准。前 30 天明确标注"系统处于冷启动期" |
| **行动建议仍可能出错** | Level 3 禁止了最危险的模式，但 Level 1 和 Level 2 的建议仍可能不适用于特定读者。系统不知道自己不知道什么 | 行动标注支撑来源，让读者自己判断。不声称行动是"建议"，只说是"信号提炼" |
| **Counter Signal 检测需要理解主流叙事** | "当日主流叙事"的归纳本身依赖 LLM 对 Alpha+Beta 的理解，可能归纳错误，导致 Counter Signal 匹配偏移 | 只在 Counter Signal 可信度足够高时才输出。不确定时空着 |
| **Theme Memory 语义匹配不精确** | 基于 LLM 的主题匹配可能将不相关事件错误关联，或遗漏同一主题的不同表述 | 定期人工审核 Theme Memory。超过 30 天不活跃的主题自动休眠 |
| **没有人工验证闭环** | 全套自动生成，没有人在回路中校验 Alpha 准度、行动合理性、Counter Signal 可信度 | 长期应加入"人工抽查 + 反馈标注"环节。当前阶段接受不完美 |
| **冷启动的 Theme Memory** | 前 7-14 天主题检测不稳定，新颖度评分偏向高估 | 前 14 天日报标注"系统冷启动期，主题追踪数据有限" |

### 8.3 一句话总结

**v2.1 把系统从"容易高置信度犯错"降低到"知道自己不确定什么"**——不可逆性被删掉，Actionability 降权，行动被分级，Counter Signal 提供对冲视角，Theme Memory 提供连续性。剩下的局限（源偏见、评分噪声、缺乏人工验证）是目前技术栈和成本约束下的结构性限制，短期无法消除但可以被标注和管理。
