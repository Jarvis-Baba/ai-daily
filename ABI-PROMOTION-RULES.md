# ABI-PROMOTION-RULES — 对象生命周期治理规则

**Status**: FROZEN v1.1
**Date**: 2026-06-03
**Scope**: 所有 Compiler 层之间的数据交换对象
**Relation**: 本规则定义"什么值得成为 ABI"以及"什么应该不再成为 ABI"。EVIDENCE-ABI 是本规则的第一份产出。
**v1.1 变更**: 新增 §9 降级规则、§10 生命周期、§11 反馈回路指标。

---

## §0 核心原则

**ABI 描述模块之间如何交换对象。它不是概念模型，不是未来规划，不是架构美学。**

一个对象只有在生产链路中被观察到满足晋升条件后，才允许拥有独立 ABI。同样，一个对象在满足降级条件后，必须被降级或移除。

```
先运行 → 观察 → 证明 → 抽象 → 冻结 → 监控 → 降级/退役
```

禁止反向：~~先设计 → 先抽象 → 先建层 → 等待使用~~
禁止永久：~~一旦成为 ABI 就永远是 ABI~~

---

## §1 晋升条件（Promotion Criteria）

一个概念必须同时满足以下 4 条中的**至少 3 条**，才允许升级为独立 ABI：

| 条件 | 含义 | 验证方式 |
|------|------|---------|
| **P1. 独立生命周期** | 对象可以在其创建者销毁后继续存在 | 删除消费该对象的 Article/Report，对象是否仍然有效？ |
| **P2. 跨上层对象复用** | 同一对象被多个不同类型的上层对象引用 | 是否被 ≥2 种不同 Compiler 或 ≥3 篇不同文章引用？ |
| **P3. 需要独立治理** | 对象需要自己的验证、审计、冲突处理规则 | 是否有独立于消费场景的质量/一致性问题需要解决？ |
| **P4. 需要独立存储/检索** | 对象需要被跨文章搜索、索引、去重 | 是否出现"找出所有关于 X 的 Y 对象"的查询需求？ |

**评分**：4/4 = 明确晋升。3/4 = 可晋升（记录理由）。≤2/4 = 不晋升，保留为上层对象的内嵌字段。

---

## §2 不晋升判定（Anti-Promotion Gate）

如果晋升条件不满足，对象保留为上层对象的**内嵌字段**。此时：

- 可以在上层 ABI 中预留 `xxx_refs: []` 作为未来接口
- 文件可以直接存储（如图片文件），但不在 ABI 层定义独立对象类型
- 不创建独立 Manager / Registry / Resolver / Store

**"预留接口 ≠ 独立 ABI"。** `media_refs` 是一个接口槽位。MEDIA-ABI 只有通过 §1 晋升条件后才存在。

## §4 候选观察名单（Candidate Watchlist）

以下对象已在 IR 中反复出现，但尚未满足晋升条件。**记录它们，观察它们，不在今天给它们 ABI。**

| 候选对象 | 当前形式 | 观察信号 | 晋升风险 |
|---------|---------|---------|---------|
| **Media Asset** | `media_refs: []`（EVIDENCE-ABI §3 预留接口） | 同一张截图被 ≥3 篇文章引用；需要独立版权审计；出现"搜索所有官方产品截图" | 过早晋升会导致只有 17 个对象却有 4 个 Manager |
| **Structural Shift** | IR 中的 `structural_shifts[]`（mechanism+trigger+consequence） | 同一个 Shift 被 ≥10 篇日报引用；被研究报告引用；出现跨 Shift 比较需求 | 可能是第一个晋升的"研究型对象"——但目前属于 L2 Content，未进入 Evidence Layer |
| **Signal** | IR 中的 `signal_map`（synthesis+causal_chain） | 同一信号被多次消费；出现信号强度追踪需求 | 比 Shift 更偏分析层，晋升门槛应该更高 |
| **Theme** | IR 中的 `active_themes[]`（agent_stack/anthropic_scaling/regulation） | 主题被跨月追踪；出现主题演化图 | 目前只是标签，不是对象 |
| **Narrative** | 不存在于 IR，但 Content Compiler 产出中隐含 | 同一叙事框架被多篇文章复用 | 可能永远不需要 ABI——它是 Content 的产物，不是输入 |

## §5 晋升流程

当候选对象疑似满足 §1 条件时：

1. **手工验证**：拿出 ≥3 个真实生产案例，逐条对照 P1-P4
2. **撰写晋升提案**：一页纸，列出四条条件的证据
3. **冻结 ABI v1**：≤200 行，含对象 Schema + 必填字段 + 禁止字段 + 版本兼容规则
4. **在下一个 Milestone 中验证**：同 M1 方法论——用真实链路证明 ABI 成立

任何跳过步骤 1 直接进入步骤 3 的行为，视为违反本规则。

## §6 反模式清单

以下行为被明确禁止：

- ❌ "未来可能需要" → 立即设计 ABI
- ❌ "这个概念很优雅" → 直接建层
- ❌ "其他系统都有这个层" → 我们也建一个
- ❌ 对象数量 < 20 但创建了独立的 Manager/Registry/Resolver
- ❌ 在 Evidence Package #010 之前冻结第二份独立 ABI（除非 §1 条件被压倒性满足）

## §9 降级规则（De-Promotion Rules）

**如果一个 ABI 不再满足存在条件，它必须被降级或移除。** 抽象自膨胀是系统熵增的主要来源。晋升没有降级对冲，系统会不可逆地向复杂化漂移。

### 降级条件

一个已冻结的 ABI，满足以下**任一条**，触发降级评审：

| 条件 | 含义 | 阈值 |
|------|------|------|
| **D1. 使用频率持续下降** | 该 ABI 在最近的 Package 中被引用的频率显著降低 | 最近 30 天内引用次数 < 早期月均的 20% |
| **D2. 不再被多上层对象复用** | 该 ABI 的消费者从多个 Compiler 退化为单一 Compiler | 连续 10 个 Package 仅被 1 种 Compiler 引用 |
| **D3. 被更高层对象吸收** | 该 ABI 的结构和字段已被更高层 ABI 原生支持，独立存在不再必要 | 两个 ABI 的字段重合度 > 70% 且上层 ABI 的消费者已覆盖下层 ABI 的消费者 |
| **D4. 对象实例数极少** | 该 ABI 在生产中只有极少实例，说明"独立对象"的假设不成立 | 该类型对象总数 < 20，且最近 30 天新增 < 3 |
| **D5. 治理成本超过使用价值** | 维护该 ABI 的版本兼容性、验证规则、迁移脚本的成本已经超过它带来的工程收益 | 需人工判定，需在降级提案中量化成本 |

### 降级处置选项

| 选项 | 含义 | 适用场景 |
|------|------|---------|
| **降级为内嵌字段** | ABI 文档归档，Schema 合并到上层 ABI 的可选字段区 | D2, D3 |
| **合并到相邻 ABI** | 两个 ABI 合并为一个，旧 ABI 标记为 `MERGED` | D3 |
| **退役（Retire）** | ABI 标记为 `RETIRED`，不再接收新实例，已有实例保留只读 | D1, D4 |
| **删除（Remove）** | ABI 及相关对象全部清理 | D5, 且确认无活跃消费者 |

### 降级流程

1. 任一条件触发 → 在 Candidate Watchlist 中标记 `⚠️ Review`
2. 人工确认降级条件非误判（例如下降是因为系统停用，而非对象本身失效）
3. 选择降级处置选项，记录理由
4. 更新相关 ABI 文档状态为 `DEPRECATED` / `MERGED` / `RETIRED`
5. 给下游消费者 ≥30 天的迁移窗口期

### 反-反模式

以下理由**不能**作为拒绝降级的依据：

- ❌ "将来可能还会用到" → 等将来再晋升。一个退役的 ABI 可以被重新激活。
- ❌ "它设计得很好" → 设计质量 ≠ 使用价值。好的设计不被使用，更说明它解决的不是真实问题。
- ❌ "删除它很麻烦" → 说明耦合过高，正是需要降级来暴露的架构问题。

## §10 ABI 生命周期（Full Lifecycle）

一个对象从被观察到被移除，经历 6 个阶段：

```
┌──────────┐
│ OBSERVE  │  在 IR/生产链路中出现，进入 §4 候选观察名单
└────┬─────┘
     ↓
┌──────────┐
│  PROVE   │  满足 §1 晋升条件 ≥3/4，用真实数据证明独立存在必要
└────┬─────┘
     ↓
┌──────────┐
│ PROMOTE  │  冻结 ABI v1.0，不超过 200 行，遵循 §5 流程
└────┬─────┘
     ↓
┌──────────┐
│STABILIZE │  通过 M1 风格端到端验证，在生产中跑 ≥3 个 Package
└────┬─────┘
     ↓
┌──────────┐
│ MONITOR  │  指标被 §11 反馈回路追踪。正常状态：此阶段持续运行。
└────┬─────┘
     ↓ (如果触发 §9 降级条件)
┌──────────┐
│DE-PROMOTE│  降级 / 合并 / 退役 / 删除
└──────────┘
```

**MONITOR 是稳态。** 一个健康的 ABI 大部分时间在 MONITOR 阶段。OBSERVE/PROVE/PROMOTE 是出生期，STABILIZE 是验证期，DE-PROMOTE 是终结期。如果 DE-PROMOTE 发生频率远低于 PROMOTE，说明系统正在不可逆地复杂化。

## §11 反馈回路指标（Feedback Loop Metrics）

### 每个 ABI 必须追踪

| 指标 | 计算方式 | 告警阈值 |
|------|---------|---------|
| **活跃消费者数** | 最近 30 天引用该 ABI 的不同 Compiler 类型数 | < 2 → 触发 D2 评审 |
| **对象实例数** | 该类型对象的总数和近 30 天新增数 | 总数 < 20 且近 30 天新增 < 3 → 触发 D4 评审 |
| **跨 Package 引用率** | 被 ≥2 个 Package 引用的对象占比 | < 30% → 触发 D2 评审 |
| **单消费者占比** | 仅被 1 种 Compiler 引用的对象占比 | > 80% → ABI 分层可能过细 |
| **无引用天数** | 该 ABI 最近一次被新 Package 引用距今的天数 | > 60 天 → 触发 D1 评审 |
| **字段实际使用率** | ABI 定义的字段中，在生产 Package 中实际被填充的比例 | < 50% → Schema 可能过度设计 |

### 每个候选对象必须追踪

| 指标 | 含义 |
|------|------|
| **出现频率** | 最近 30 天在 IR/生产链路中出现的次数 |
| **首次出现日期** | 用于计算"候选时长"。候选 > 90 天未晋升 → 重新评估是否需要独立 ABI |
| **结构稳定性** | Schema 在最近的观察中是否还在变化。不稳定 → 不具备晋升条件 |

### 指标的使用方式

这些指标**不是自动决策系统**——它们是触发人工评审的信号。指标达标 + 人工确认 = 行动。

没有指标的人工决策是印象。没有人工的自动指标是盲人。

### Epistemic Extension Candidates（认知扩展候选）

以下维度在当前系统中被识别为结构性缺口，但尚未观察到因缺失导致的决策失败。**记录但不实现。** 等待系统压力触发后再进入 ABI review。

| 缺口 | 描述 | 当前缓解方式 | 触发条件 |
|------|------|------------|---------|
| **domain_authority** | source_reliability 衡量来源的一般可信度，但不衡量来源在特定 topic 上的发言权。同一来源在 AI safety、benchmarking、economics 三个话题上的 authority 可能完全不同。 | 目前仅 Anthropic 一个官方来源，authority 差异尚未暴露。M1.5 的 D-002 被正确地保留为 unverified 而非升级为 disputed——说明现有模型在"不确定时"选择了保守策略，这是正确的。 | 出现 ≥1 个高置信度 disputed cluster（≥3 个来源冲突，且按现有 confidence 模型无法判定哪方更可信） |
| **epistemic convergence** | 当 Evidence 冲突时（§5），L2 Content Compiler 如何形成最终输出结论。当前 §5 只定义"保留冲突，交给 L2"，但 L2 的裁决规则（依据什么选择、如何标注不确定性）未定义。这不是 ABI 字段问题，是 L2 的决策流程缺失。 | 尚未出现需要裁决的 disputed case。M1.5 的 D-002 是潜在 disputed 而非确认 disputed——系统在"不确定"时选择了不输出争议结论，这本身是安全的保守策略。 | 同一 disputed Evidence 在不同文章中产生了不一致的结论（即：同一组 Evidence → 两篇不同的 L2 输出给出了不同的裁决结果） |

**原则**：这些缺口是被观察到的，不是被预判的。它们留在 PROMOTION RULES 中而非进入任何 ABI，因为——结构缺口开始影响决策之前，不升级模型。

## §8 冻结声明

本规则定义 ABI 的完整生命周期治理标准：晋升（§1）、候选观察（§4）、降级（§9）、反馈回路（§11）。

**演化原理**：本规则自身也遵守它所定义的规则。如果未来出现"对象治理系统中的对象需要治理"的需求，不排除从本规则中提取出独立的 GOVERNANCE-ABI。目前不需要。

修改本规则需满足：
- 至少 1 个新对象通过了 §5 晋升流程，或 1 个已有 ABI 触发了 §9 降级流程
- 流程中发现了本规则的缺陷

**下一评审窗口**：发生第一次晋升或第一次降级后。
